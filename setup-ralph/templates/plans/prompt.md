# ISSUES

Issues JSON is provided at start of context. Parse it to get all issues with their state, bodies, and comments.

Treat open GitHub issues as the source of truth for tracked work.

- Open issues are candidate work.
- Closed issues are not candidate work unless they are reopened.
- Do not recreate work from closed issues unless a new open issue explicitly asks for it.
- Do not assume a closed issue was correctly closed. For reopened issues, trust the reopened state and the current acceptance criteria.

You've also been passed a file containing the last 10 RALPH commits (SHA, date, full message). Review these to understand what work has been done.

# TASK BREAKDOWN

Break down the issues into tasks. An issue may contain a single task (a small bugfix or visual tweak) or many, many tasks (a PRD or a large refactor).

Make each task the smallest possible unit of work. We don't want to outrun our headlights. Aim for one small change per task.

If an issue is a parent PRD or umbrella issue, use its open child issues, unchecked checklist items, and explicitly referenced open blockers as the only tracked remaining work.

Do not invent new work from vague polish ideas, future nice-to-haves, or "next likely slice" notes in comments when that work is not tracked by an open issue or unchecked checklist item.

# TASK SELECTION

Pick the next task. Prioritize tasks in this order:

1. Critical bugfixes
2. Tracer bullets for new features

Tracer bullets comes from the Pragmatic Programmer. When building systems, you want to write code that gets you feedback as quickly as possible. Tracer bullets are small slices of functionality that go through all layers of the system, allowing you to test and validate your approach early. This helps in identifying potential issues and ensures that the overall architecture is sound before investing significant time in development.

TL;DR - build a tiny, end-to-end slice of the feature first, then expand it out.

3. Polish and quick wins
4. Refactors

If all tracked tasks are complete, output <promise>COMPLETE</promise>.

A parent PRD or umbrella issue is metadata unless it contains its own concrete, still-unverified acceptance criteria for the currently tracked scope.

A parent PRD or umbrella issue should be treated as complete for loop-termination purposes when all of the following are true:

1. All linked or referenced child issues are closed.
2. There are no unchecked checklist items remaining in the parent issue.
3. There are no explicitly referenced open blocker issues remaining.
4. There is no failing validation that prevents the tracked scope from being considered done.
5. The remaining parent issue text is only broad product context, future ideas, or explanatory prose rather than a new concrete tracked task.

When those conditions are met:

- do not mine the PRD text for more optional polish,
- do not keep iterating just because the parent PRD issue itself is still open,
- treat closing the parent PRD or umbrella issue as the final remaining task,
- after confirming there is no remaining open child issue and no unverified acceptance criterion, update the parent issue if needed, leave a concise verification comment, and close it,
- then output <promise>COMPLETE</promise> if no other tracked task remains.

After closing or completing any tracked child issue in an iteration, immediately re-evaluate the remaining open tracked issues before ending that iteration.

If that re-check shows that the only remaining open tracked issue is the parent PRD or umbrella issue, and the parent meets the completion conditions above, do not wait for a future iteration:

- close the parent in the same iteration,
- emit <promise>COMPLETE</promise> in the same iteration,
- do not consume another loop iteration just to discover that only the parent remains open.

# FEEDBACK LOOPS

Before committing, run the feedback loops:

- npm run test to run the tests

Then run task-specific verification against the issue's acceptance criteria.

- If the issue is about UI, UX, layout, navigation, visibility, responsive behavior, or persona-specific behavior, verify the actual rendered behavior in the running app.
- Verify with the relevant persona(s) named in the issue or PRD.
- Do not treat source code, a commit, or passing unit tests as sufficient proof that a UX issue is done.
- If you cannot verify a criterion directly, leave it unchecked and keep the issue open.

# THE ISSUE

An issue is complete only when every acceptance criterion in the issue body has been explicitly verified.

Before closing any issue:

1. Re-read the acceptance criteria.
2. Verify each criterion with concrete evidence from the running app, tests, or other direct validation.
3. Update the issue body checklist so verified criteria are [x] and unverified criteria stay [ ].
4. Leave a comment summarizing exactly how each criterion was verified.

Never close an issue based only on any of the following:

- code existing in the repo
- a commit being created
- a comment claiming completion
- partial implementation
- tests that do not exercise the user-visible acceptance criteria

If every acceptance criterion is verified, close the original GitHub issue.

If you completed a child issue, close that child issue only after updating its checklist and leaving the verification comment.

If you determine that a parent PRD or umbrella issue is complete because all tracked child issues, checklist items, and blockers are done, close the parent issue only after confirming there is no remaining open child issue and no unverified acceptance criteria.

If the task is not complete, leave a comment on the GitHub issue with what was done, what remains, and which acceptance criteria are still not satisfied.
